## 更新后的大纲

### 标题（v14）

**LLM 判官与领域专家的系统性分歧：ISO 15189:2022 医学实验室文件生成的多层次评估研究**

（英文备选：Systematic Divergence Between LLM Judges and Domain Experts in LLM-Assisted ISO 15189:2022 Medical Laboratory Document Generation: A Multi-Level Validation Study）

### 摘要（v14）

本研究针对 LLM 辅助医学实验室质量管理体系（QMS）文件生成的评估方法学问题，首次构建 **三层正交评估范式**——自动合规评分（Python scorer）+ 双 LLM judge（Claude Opus 4.6 + GPT-5.4）+ 领域专家盲评（CNAS 资格检验科主任技师）。通过 **2×2 对称生成设计**（两生成模型 × 9 组配置 × 15 任务 × 3 重复 = 486 篇 QMS 文件），在 LLM 层获得 864 次评分；在专家层对 10 篇分层随机样本（覆盖 7 组 × A 类"程序"任务）进行盲评。主要发现：

1. **LLM 判官与专家评估存在系统性分歧**：
   - ICC(3,1) 专家 vs Claude judge = **0.613** (moderate, p=0.02)
   - ICC(3,1) 专家 vs GPT judge = **0.195** (poor, ns)
   - 两个 LLM 判官都系统性**高估** 0.55–0.93 分

2. **排名方向性分歧**：
   - LLM 判官：H4_sop_only (6K) 综合最优
   - **专家**：G_template_rules (38K) ≈ F_template (35K) 最优；**H4 在专家视角倒数第二**（仅优于 A_bare 基线 0.16 分）

3. **4 因子正交消融在 LLM 层依然成立**（规则层有效、骨架层有效、详细内容/示例无贡献），但**在专家层揭示 token 效率与临床可用性的权衡**：LLM 判官对"结构完整性"敏感，专家对"临床实操细节"（急诊样本、拒收标准、运送人员资格认证等）敏感

4. **Claude 判官的排序能力可作为专家的粗略代理**（ρ=0.70），但 GPT 判官不具备此代理能力（ρ=0.30）

5. **C_full (71K) 在 GPT 生成下崩溃**（跨评 1.40–1.84）的 v13 发现在 v14 仍成立，进一步说明 token 体量陷阱的模型依赖性

6. **实践建议（场景分层）**：
   - 初稿/内部草稿/快速迭代 → **H4 或 E_rules**（3–6K，效率优先）
   - 正式文件/CNAS 送审/临床实操 → **G_template_rules 或 F_template**（35–38K，临床可用性优先）
   - 避免使用 → **C_full (71K)**（跨模型崩盘）与 **A_bare (0K)**（基线太低）

7. **方法学贡献**：提出"三层正交评估 + 场景分层推荐"框架，替代"寻找单一最优配置"的研究范式

### 1. 引言

#### 1.1 背景与动机
- ISO 15189:2022 → CNAS-CL02:2023 推动文件体系重建
- LLM 辅助专业文档生成的评估方法学缺口：
  - 现有研究依赖单一 LLM judge（引入自评偏见）
  - 或依赖 token-level 自动指标（缺乏领域效度）
  - **均缺乏领域专家验证层**
- 本研究首次将 **领域专家盲评**作为三层评估中的"真值锚"

#### 1.2 研究问题（v14 更新）

- **RQ1**：配置文件中哪些成分对 LLM 输出质量真正有贡献？（4 因子消融）
- **RQ2**：领域知识的"内容"和"结构"哪个更重要？
- **RQ3**：示例内容是帮助 LLM 学习还是让 LLM 照抄？
- **RQ4**：最精简的有效配置是什么大小？
- **RQ5**：跨生成模型（Claude vs GPT）下最优配置是否一致？
- **RQ6**：LLM judge 的自评偏见有多大？能否通过 2×2 对称设计完全量化？
- **RQ7（v14 新增）**：**LLM 判官的评估是否与领域专家评估一致？若不一致，分歧的模式是什么？**

#### 1.3 贡献声明（v14 更新）

1. **方法学原创**：首次构建"**自动合规评分 + 双 LLM judge + 领域专家盲评**"的三层正交评估范式，填补 LLM 配置研究缺乏专家校验层的空白
2. **核心实证发现**（反 v13）：
   - LLM 判官偏好极简配置（H4, 6K），但**专家偏好模板化配置**（G/F, 35–38K）
   - Claude 判官具有专家排序代理能力（ρ=0.70, ICC=0.61），GPT 判官不具备（ρ=0.30, ICC=0.20）
   - LLM 判官系统性高估 0.55–0.93 分
3. **4 因子正交消融量化各成分贡献**（v13 成果延续，在 LLM 层有效）：
   - 规则层：+0.511***；骨架层：+0.213 p=0.055
   - 详细内容：无贡献；示例：轻微有害
4. **首次精确量化 Claude 与 GPT 的自评偏见**：+0.464 vs −0.472
5. **"token 体量陷阱"的模型依赖性**：C_full 在 Claude 生成下勉强可用，在 GPT-5.4 生成下彻底崩溃（跨评 1.40–1.84）
6. **场景分层实践指南**：替代"单一最优"推荐
7. **开放数据**：486 篇生成、864 次 LLM 评分、10 篇专家盲评、ICC 分析脚本全部公开

（略，同 v13）

#### 3.5 生成模型

- **主实验（Claude Opus 4.6）**：9 组 × 15 任务 × 3 重复 = **405 篇**
- **2×2 对称验证（GPT-5.4）**：9 组 × 3 任务（A1/B1/C1）× 3 重复 = **81 篇**

#### 3.6 三层评估体系（v14 核心章节）

**层 1：自动合规评分**
- Python scorer：格式/条款/术语/违规/模糊，486 篇全覆盖
- 术语映射表经领域专家校验

**层 2：双 LLM Judge**
- Claude Opus 4.6 judge（CNAS 主任评审员 prompt，5 维度）
- GPT-5.4 judge（同 prompt 跨模型验证）
- 覆盖：378 篇（270 Claude-gen + 27 H-gen + 81 GPT-gen）+ 486 篇 = 864 次评分
- **2×2 对称设计**：Claude-gen → Claude-judge / Claude-gen → GPT-judge / GPT-gen → Claude-judge / GPT-gen → GPT-judge

**层 3：领域专家盲评（v14 新增）**
- 专家：检验科主任技师，CNAS 资格，ISO 15189 评审经验
- 样本：10 篇分层随机采样（seed=42），覆盖 7 组（H4×2, C_full×2, G×2, H2×1, E×1, A×1, F×1）× A 类"程序"任务（A1/A2/A3）
- 盲化：随机打乱 paper_01..10 编号，去除组名标识；专家不知道哪篇来自哪组
- 评分维度：与 LLM judge 完全一致的 5 维度，每维度 0–5 分（允许小数）
- 盲评理由：专家可选标注具体缺陷
- 结果分析：Pearson/Spearman/ICC(2,1)/ICC(3,1) vs 两个 LLM judge

#### 3.8 统计方法

- Kruskal-Wallis + Mann-Whitney U + Bonferroni
- 2×2 因子效应量（双因子 ANOVA）
- 跨 judge 一致性：Pearson、Spearman、ICC(2,1)/ICC(3,1)
- 偏见量化：跨模型差分统计
- **专家-LLM 一致性**：ICC(2,1) 绝对一致、ICC(3,1) 排序一致、Bland-Altman 差异分析

#### 3.9 专家盲评方法细节（v14 新增）

**抽样策略**（分层随机）：
- 固定种子 42，按 A 方案抽样
- 组分布：H4×2（验证"最优"声明）、C_full×2（验证跨评崩盘）、G×2（验证跨评第一）、H2/E/A/F 各 1
- 任务范围：仅"程序"类（A1/A2/A3），避免 B/C 类审核报告与检查表对 5 维度适配性差

**盲化实施**：
- 脚本 `prepare_blind_review.py` 去除组名标识
- 随机打乱 paper_NN 分配
- 专家评分表 `rating_sheet.md` 仅显示任务类型（如"人员培训与能力评估控制程序"），不显示组名

**Claude judge H 组数据补齐**：
- 原 Claude judge 对 H 组（H2/H3/H4）未保留逐篇数据
- 通过 Claude sub-agent 用完全一致的 CNAS 评审 prompt 补跑 3 篇 H 组（2026-04-14）

**ICC 计算**：
- 使用 pingouin 库
- ICC(2,1)：two-way random, absolute agreement, single rater
- ICC(3,1)：two-way mixed, consistency, single rater
- 结合 95% CI 和 p 值报告

#### 5.5 Token 效率 vs 临床可用性的权衡（v14 新增核心章节）

**核心洞察**：LLM 层的"最优"和专家层的"最优"反映了**不同的质量维度**。

**LLM 判官的评估偏好**（**"内容表面完整性"**）：
- 章节结构完整 → 高分
- 术语合规 → 高分
- 表单编号列表完整 → 高分
- **结果**：骨架版（H4, 13K）能以最小代价满足上述三项，**token 效率最高**

**专家的评估偏好**（**"临床实操可用性"**）：
- SOP 步骤是否覆盖所有临床子场景（如急诊、拒收、患者自采）→ 权重最高
- 责任岗位是否明确到具体人员资格要求 → 权重高
- 记录表单是否在真实工作流中可追溯 → 权重高
- **结果**：模板版（G, 38K）包含丰富临床细节而显著优于骨架版

**权衡框架**：

| 维度 | H4 (6K) | G (38K) | 权重来源 |
|------|---------|---------|---------|
| 章节结构完整性 | ✅ 高 | ✅ 高 | LLM judge 主导 |
| 术语规范 | ✅ 高 | ✅ 高 | LLM judge 主导 |
| 临床子场景覆盖 | ❌ 低 | ✅ 高 | 专家主导 |
| 岗位资格具体性 | ❌ 中 | ✅ 高 | 专家主导 |
| 记录可追溯性 | ❌ 中 | ✅ 高 | 专家主导 |
| **token 开销** | **低** | **高** | 成本主导 |

**推论**：**不存在"单一最优"配置；只存在"场景匹配的最优"配置**。

#### 5.6 为什么 Claude judge 比 GPT judge 更接近专家？（v14 新增）

**可能的解释**（待后续研究验证）：

1. **训练数据的领域覆盖**：Claude 的训练数据可能包含更多中文医疗/质量管理文本，使其对中文 ISO 文件的评估更接近人类专家
2. **指令跟随精度**：Claude 对"CNAS 主任评审员视角"的角色代入更深，而 GPT 可能趋于"通用文档质量评估"
3. **评分粒度**：Claude 在 0–5 整数刻度上展现更细的分辨力（均值 4.4–4.9），GPT 趋于"中庸化"（均值多聚集 3.8–4.3）

### 6. 讨论

#### 6.1 三层评估范式的价值（v14 重写）

**传统单层评估的盲点**：
- 仅自动评分：缺乏语义理解
- 仅 LLM judge：受自评偏见和"表面特征偏好"污染
- 仅专家评分：样本量受限、一致性难保证

**三层互补的优势**：
- 自动评分提供**客观基线**
- LLM judge 提供**规模化扩展**
- 专家评分提供**领域效度锚**
- **ICC 对比三层结果**，可诊断评估方法的系统偏差

#### 6.2 LLM 判官偏好的"表面特征效应"（v14 核心讨论）

**新命名的现象**：**"表面完整性偏见"**（surface-completeness bias）

**证据链**：
1. Claude judge 偏见 +0.464（v13）→ 偏向"内容丰富感"
2. 专家对 H4 评 3.06，LLM 判官评 4.20–4.40 → 偏差 1.14–1.34 分
3. 专家注明的缺陷（急诊样本、拒收标准、运送人员资格）都是**条款覆盖的"深度"而非"广度"**
4. LLM judge 擅长验证"广度"（章节/表单/术语），不擅长验证"深度"（实操可执行性）

**对 LLM-as-Judge 研究的贡献**：
- 首次通过专家层验证，识别**LLM 判官的"表面完整性偏见"**
- 建议未来的 LLM judge prompt 设计应**显式提示"临床实操深度"评估**

#### 6.3 配置精简化原则的更新（v14）

**v13 原则**：只保留规则+骨架，精简至 6K
**v14 原则（更新）**：
- **如果评估标准是 LLM 判官**（或自动指标）：精简至 6K 足够
- **如果评估标准是临床可用性**：需保留模板层（35K），不可精简
- **结论**：**配置的"最优"取决于评估标准；不存在绝对最优**

#### 6.4 自评偏见的方法论启示（延续 v13）

| Judge | 偏见均值 | 性质 |
|-------|---------|------|
| Claude | +0.464 | 正向偏见（偏爱自生成） |
| GPT | −0.472 | 反向偏见（偏爱 Claude 生成） |

**骨架版配置（E/H3/H4）的偏见最小（<0.15）**——但这并不代表它们在专家视角下最优（v14 专家层发现推翻了 LLM 层的"最优"结论）

#### 6.5 场景分层实践指南（v14 重写核心）

**替代 v13 的"单一推荐 H4"**，v14 按场景分层：

| 使用场景 | 推荐配置 | Token | 依据 | 权衡 |
|---------|---------|-------|------|------|
| **快速生成初稿** | H4_sop_only | 6K | 效率最优、LLM 判官满意 | 需人工补充临床细节 |
| **内部草稿/迭代** | E_rules + H3_skeleton | 3–13K | 成本低，便于快速比对 | 同上 |
| **正式文件（CNAS 送审）** | **G_template_rules** | **38K** | 专家评分最高、跨模型稳健 | token 成本高 |
| **完整质量手册** | F_template 或 G | 35–38K | 临床细节最丰富 | 同上 |
| ❌ 不推荐 | C_full (71K) | — | GPT 生成下崩盘、跨评最差 | — |
| ❌ 不推荐 | A_bare/B_simple | — | 基线太低、缺合规性 | — |

**核心建议**：
- **追求 token 效率**：H4（6K）→ 人工校对补充临床细节
- **追求合规通过**：G（38K）→ 接受更高 token 成本换取专家认可

#### 6.6 LLM 判官与专家分歧的实践警示（v14 新增）

**对实验室/医院管理层的建议**：
- **不要**仅依赖 LLM 判官评分做配置决策
- **应该**在关键文件（管理评审/认可送审）生成后至少通过领域专家抽查验证
- **可以**使用 Claude judge 做**初筛**（ρ=0.70 代理专家），但**必须**后续专家终审

**对研究社区的建议**：
- LLM-as-Judge 方法应明确声明评估的是"表面质量"而非"深度质量"
- 专业领域（医疗/法律/金融）的 LLM 评估研究**必须包含领域专家层**
- 未来工作应研究如何让 LLM judge 更接近专家（如 Chain-of-Thought prompt, 专家标注微调）

### 7. 局限性（v14 更新）

- **7.1 专家样本有限**：仅 1 位专家对 10 篇评分，每组 n=1–2。需扩展至 3–5 位专家 × 30–50 篇以估计 Fleiss κ 和专家间一致性
- **7.2 专家与研究者身份重合**：本研究的专家评分者同时是通讯作者，存在潜在偏见风险。未来应招募独立专家
- **7.3 跨模型验证样本受限**：GPT-5.4 生成仅覆盖 A1/B1/C1 三任务（81 篇）
- **7.4 领域特异性**：针对 ISO 15189 医学实验室，对 ISO 17025/ISO 9001 的迁移性待验证
- **7.5 骨架化算法的主观性**：strip_to_skeleton 规则基于作者判断
- **7.6 SOP 选择依赖领域专家**：H4 的任务→SOP 映射非自动化
- **7.7 第三方 judge 缺失**：仅 Claude/GPT 两家，Gemini/open-source 模型未覆盖
- **7.8 语言限定**：全部中文
- **7.9 评估维度的效度**：5 维度 prompt 是否充分覆盖 CNAS 审评的实际关注点需进一步验证

### 8. 结论（v14 重写）

1. **LLM 判官与领域专家在 ISO 15189 文件生成评估中存在系统性分歧**：
   - Claude judge 与专家排序中等一致（ICC(3,1)=0.613, p=0.02），但系统性高估 0.93 分
   - GPT judge 与专家排序相关弱（ICC(3,1)=0.195），系统性高估 0.55 分
   - **LLM 判官不可替代领域专家做最终评估**

2. **配置架构的"最优"取决于评估标准**：
   - **LLM 判官视角**：H4_sop_only (6K) 最优（token 效率最高）
   - **专家视角**：G_template_rules (38K) ≈ F_template (35K) 最优（临床可用性最高）
   - **H4 在专家排名中倒数第二**，与 LLM 层评估颠覆性分歧

3. **4 因子正交消融揭示配置成分的贡献方向性**（LLM 层）：
   - 规则层：必需（+0.511***）
   - 骨架层：必需（+0.213, p=0.055）
   - 详细内容：LLM 层无贡献（Δ=−0.031），**但专家层有显著贡献**
   - 示例：LLM 层轻微有害，专家层未单独验证

4. **Token 体量陷阱的模型依赖性**：C_full (71K) 在 Claude 生成下跨评 3.22–4.56，在 GPT 生成下跨评 1.40–1.84。**"全量配置可用"是 Claude 特性，非通用结论**

5. **2×2 对称设计精确量化了 LLM judge 的自评偏见**：Claude +0.464 / GPT −0.472。骨架版配置（E/H3/H4）的偏见最小（<0.15），但这种"稳健性"只在 LLM 层有意义，在专家层这些配置反而排名靠后

6. **实践建议（场景分层）**：
   - 快速生成初稿 / 内部迭代 → **H4 或 E_rules**（6K 左右，效率优先）
   - 正式文件 / CNAS 送审 / 临床执行 → **G_template_rules 或 F_template**（35–38K，专家认可度优先）
   - 避免 → C_full (71K)（跨模型崩盘）、A_bare (0K)（基线太低）
   - **所有配置生成的关键文件必须通过领域专家终审**

7. **方法学原则**：
   - 专业领域 LLM 评估**必须包含领域专家层**
   - LLM judge 的排序可作为**初筛**（Claude 代理能力 ρ=0.70），但**不可作为终审**
   - "表面完整性偏见"是 LLM judge 的系统性缺陷，未来评估设计应显式提示"深度评估"
