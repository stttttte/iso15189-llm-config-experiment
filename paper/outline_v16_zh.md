## 更新后的大纲

### 标题（v16）

**"LLM 在不同配置下生成医学实验室 QMS 文件的合规性研究"**

（英文备选：A Compliance Study of LLM-Generated Medical Laboratory QMS Documents Under Different Configurations）

### 摘要（v16 更新）

本研究通过 9 种配置的对比实验，探讨给 LLM 什么样的配置能生成最合规、最可用的 ISO 15189:2022 医学实验室质量管理体系（QMS）文件。实验设计：**2 个生成模型**（Claude Opus 4.6 + GPT-5.4）× **9 组配置**（无配置 → 全量配置 71K → 规则+骨架 6K 等）× 15 任务 × 3 重复，共生成 486 篇 QMS 文件，叠加 **3 层评估**（Python 自动合规评分 + 2 个 LLM judge 跨模型评估 + **3 位 ISO 15189 资格专家盲评 10 篇**）。主要实证发现：

1. **4 因子正交消融**揭示配置成分的 LLM 层贡献：规则层显著有效（+0.511***）、骨架层有贡献（+0.213, p=0.055）、**详细内容无贡献**（p=0.79）、**示例内容轻微有害**（p=0.46）

2. **全量配置陷阱具模型依赖性**：C_full (71K) 在 Claude 生成下 LLM 评分 3.22–4.56，在 GPT-5.4 生成下跨评降至 1.40–1.84——"堆料即好"是 Claude 的特殊能力，**不是通用结论**

3. **LLM 评审存在自评偏见**：Claude +0.464（偏爱自生成，8/9 组正偏）；GPT −0.472（反向偏见）。跨评最客观排名（规避自评偏见）：G_template_rules (38K) ≈ H3_skeleton (13K) ≈ E_rules_v2 (3.1K) ≈ H4_sop_only (6K)，**统计等价**

4. **3 位专家盲评（n=3 raters × 10 papers）揭示 token 效率与临床可用性的权衡**：
   - **LLM 评审视角**：H4_sop_only (6K) 综合最优
   - **专家视角**：F_template (35K) ≈ H2 ≈ G_template_rules (38K) 最优（专家均值 4.06–4.24）；**H4 在专家排名中倒数第二**（3.20 vs F/G 的 4.06–4.24）
   - **专家间一致性 ICC(2,k)=0.982**（优秀），评分维度设计高度客观
   - **专家均值 vs Claude judge ICC(3,1)=0.548** (p=0.04)；vs GPT judge ICC(3,1)=0.217 (ns)
   - LLM 评审系统性高估 0.52–0.90 分（C_full 高估最严重：Claude 4.90 vs 专家 3.45）

5. **实践建议（场景分层）**：
   - **追求 token 效率**（初稿/内部迭代）→ H4_sop_only (6K) 或 E_rules (3K)
   - **追求临床可用性**（正式送审/CNAS 申请）→ **G_template_rules (38K)** 或 F_template (35K)
   - **避免**：C_full (71K) 跨模型崩盘 / A_bare 基线太低

### 1. 引言

#### 1.1 背景与研究缺口

- ISO 15189:2022 → CNAS-CL02:2023 推动检验医学文件体系重建
- LLM 辅助 QMS 文件生成具有效率潜力，但配置策略尚未系统研究：
  - "信息量假设"（全量堆砌）vs"极简假设"（规则+骨架）的经验争议
  - **现有 LLM 配置研究**多关注模型选择或 prompt 工程的单一维度，**缺乏对 QMS 专业场景下"规则/模板/骨架/示例"多维度的系统对比**
  - 配置效果的评估多依赖单一评估方式（自动指标 或 单一 LLM judge 或 少量人类评分），**多层评估交叉验证的数据较少**

#### 1.2 研究问题

- **RQ1**：配置文件中哪些成分（规则/骨架/详细内容/示例）对 LLM 生成 QMS 文件的质量有贡献？
- **RQ2**：领域知识的"内容"和"结构"哪个更重要？ISO 原文摘录是否必要？
- **RQ3**：示例内容是帮助 LLM 学习还是让 LLM 照抄？
- **RQ4**：最精简的有效配置是多少 token？
- **RQ5**：跨生成模型（Claude vs GPT）下最优配置是否一致？
- **RQ6**：LLM judge 的自评偏见有多大？
- **RQ7**：在多层评估交叉对比下（自动 + LLM judge + 专家），哪种配置最稳健？

#### 1.3 贡献声明（v15 重写——实证发现为主）

**实证贡献**（本研究的核心）：

1. **系统比较 9 种配置**（从 0K 到 71K）在 ISO 15189 QMS 文件生成中的表现，涵盖规则/骨架/详细内容/示例四维度的正交消融
2. **跨模型验证**（2 生成模型 × 2 LLM judge）揭示 C_full (71K) 在 GPT-5.4 生成下彻底崩盘——"全量堆砌可用"是 Claude 的模型特异性现象
3. **通过 2×2 对称设计量化 LLM judge 的自评偏见**：Claude +0.464, GPT −0.472
4. **Token 效率 vs 临床可用性的权衡框架**：LLM 层最优（H4, 6K）≠ 专家层最优（F/G, 35–38K），为不同使用场景提供**分层配置建议**；专家间 ICC(2,k)=0.982 的高一致性确认该分歧非评分者噪声
5. **配置成分贡献的方向性结论**：规则和骨架必需；详细内容在 LLM 层无贡献但在专家层有价值；示例方向性有害

**方法学说明**（不声称原创，诚实引用前人）：

6. 借鉴 MedQA、ProxyQA、Chatbot Arena 等前人在专业领域 LLM 评估中的专家校验实践，将"自动评分 + 2 LLM judge + 专家盲评"的多层评估应用于 ISO 15189 QMS 场景
7. 术语表经领域专家校验修正（删除 3 条错误映射，修正 3 条目标术语）；合规评分器改进支持 9 章结构

**公开资源**：

8. 486 篇生成文件 + 864 次 LLM 评分 + 10 篇专家盲评 + 全部分析脚本开放

### 3. 方法

#### 3.1 实验配置设计

**九组对照配置**（4 因子正交消融 + 经典基线）：

| 组 | 名称 | 大小 | 规则 | 骨架 | 详细内容 | 示例 |
|----|------|------|------|------|---------|------|
| A_bare | 无配置 | 0 | ❌ | ❌ | ❌ | ❌ |
| B_simple | 简单提示 | 0.4K | 部分 | ❌ | ❌ | ❌ |
| C_full | 全量倾倒 | 71K | 部分 | ✓ | ✓ | ✓ |
| E_rules | 精炼规则 | 3.1K | ✓ | ❌ | ❌ | ❌ |
| F_template | 领域模板 | 35K | ❌ | ✓ | ✓ | ❌ |
| G_template_rules | 模板+规则 | 38K | ✓ | ✓ | ✓ | ❌ |
| **H2** | **保留示例** | **61K** | ✓ | ✓ | ✓ | **✓** |
| **H3** | **模块骨架** | **13K** | ✓ | ✓ | ❌ | ❌ |
| **H4** | **SOP 骨架** | **6K** | ✓ | **部分** | ❌ | ❌ |

#### 3.2 规则层设计（经领域专家校验）

基于 rules.md（3.1K），经检验科主任技师领域专家校验修正：
- 删除 3 条错误映射（仪器→设备、校准品→校准物、室间质评→能力验证）
- 修正 3 条目标术语（标本→样品、检验项目→检测项目、线性范围→可报告范围）
- 保留 10 条正确映射 + 7 类模糊禁令

#### 3.3 骨架层设计

**骨架化算法**（`strip_to_skeleton`）：保留章节标题/首段说明/表单编号列表；删除 blockquote（ISO 原文）、详细表格、⚠️ 示例、详细步骤、代码块

#### 3.4 任务→SOP 映射（H4 专用）

（略，15 任务的 SOP 选择映射表）

#### 3.5 生成模型

- **主实验（Claude Opus 4.6）**：9 组 × 15 任务 × 3 重复 = **405 篇**
- **跨模型验证（GPT-5.4）**：9 组 × 3 任务（A1/B1/C1）× 3 重复 = **81 篇**
- 参数：temperature=1.0, max_tokens=12000, 同一 system prompt

#### 3.6 多层评估

| 层 | 评估者 | 覆盖 | 维度 |
|----|--------|------|------|
| 1 | Python 自动合规评分器 | 486 篇 | 格式/条款/术语/违规/模糊 |
| 2a | Claude Opus 4.6 judge | 378 篇 | 5 维度（CNAS 主任评审员 prompt）|
| 2b | GPT-5.4 judge | 486 篇 | 同上 prompt（跨模型验证）|
| 3 | **3 位 ISO 15189 资格专家** | 10 篇盲评 × 3 rater | 同上 5 维度，0–5 分（允许小数）|

**层 2 的 2×2 对称设计**：Claude-gen → Claude-judge / Claude-gen → GPT-judge / GPT-gen → Claude-judge / GPT-gen → GPT-judge，用于分解自评偏见

**层 3 的专家组组成**（v16 更新）：
- Rater 1：第一作者，CNAS 主任技师资格、具 ISO 15189 认可评审经验
- Rater 2：同单位合格 ISO 15189 内审员
- Rater 3：同单位合格 ISO 15189 内审员

**层 3 的盲化实施**：
- 分层随机采样（seed=42），覆盖 7 组 × A 类"程序"任务（A1/A2/A3）
- 随机打乱 paper_01..10 编号，去除组名标识；专家不知道配置组映射
- 三位专家**独立评分**，不互相查看评分或讨论；Rater 2/3 完成评分后才与 Rater 1 数据对比
- 评分前获取 Rater 2/3 的知情同意

#### 3.7 统计方法

- Kruskal-Wallis + Mann-Whitney U + Bonferroni
- 2×2 因子效应量（双因子 ANOVA）
- 一致性统计：Pearson / Spearman / ICC(2,1) / ICC(3,1)（Shrout & Fleiss, 1979）
- 偏见量化：跨模型差分（bias = own-own − cross-own）

|------|---------|---------|-----------|------------|------------------|
| **专家 vs Claude** | 0.177 | **0.548** (p=0.04) | 0.573 | 0.509 | **−0.905** |

**四大发现**：

1. **LLM 层最优（H4）与专家层最优（F/G）的排序分歧在 n=3 下依然稳健**：H4 在 LLM 综合分 #1，专家排名 **#5 / 7 组**（倒数第二），仅比 A_bare 高 0.16
2. **专家间高度一致（ICC=0.98）但专家-LLM 一致度中等/差**：Claude 中等（ICC=0.548），GPT 差（ICC=0.217）——说明 LLM judge 捕捉到的"质量"与专家定义的"质量"存在系统性差异
3. **LLM judge 对 C_full 与 H4 的高估最严重**（Δ=−1.27 / −1.05）——两者的共同点：**结构完整 + 术语合规**让 LLM 评审满意，但临床实操深度不足
4. **规则层单独的专家层贡献不显著**：E_rules_v2 (3.19) 与 A_bare (3.04) 专家均值几乎持平，说明**仅加规则层而无骨架、详细内容，对临床专家视角的提升微乎其微**

**专家标注的典型临床缺陷**（摘自 3 位 rater 的 rationale 综合）：
- paper_01 (E-A3)："没有患者自采送检的操作；运送人员资格认证；急诊标本接收；拒收标准未说明"——临床实操盲点
- paper_03 (C_full-A1)："授权管理怎么在里面"——结构性缺失被 LLM 评审漏判
- H4 两篇（paper_09/10）：专家均值 3.28/3.11，反映 6K 极简配置在临床可用性上的不足

### 5. 理论阐释

#### 5.1 规则层和骨架层为何必需（LLM 层）

规则层：提供术语约束、模糊禁令；骨架层：提供章节结构、表单编号
这两层合计 3K（E_rules）到 6K（H4），在 LLM 层已能达到中高分

#### 5.2 详细内容和示例的 LLM 层表现

**详细内容**（ISO 原文摘录、详细表格）：LLM 层 Δ≈0（p=0.79）→ LLM 自身已内化 ISO 条款
**示例**（"⚠️ 张医生 2025-03-01 完成培训"）：LLM 层 Δ=−0.053（p=0.46）→ LLM 倾向照抄示例而非创新

**跨 judge 稳健性**：示例在 Claude judge 方向性无害，在 GPT judge 方向性有害——跨评均值下 H2 < G

#### 5.3 SOP 级 vs 模块级（H4 vs H3）

LLM 层 Δ=−0.009（ns）→ **任务特定 SOP 骨架（6K）= 完整模块骨架（13K）**（仅 LLM 层）

#### 5.4 C_full 的模型依赖崩盘

Claude Opus 的 long-context attention 能从 71K 中筛选信号，GPT-5.4 在长配置下出现指令遗忘、结构错乱、内容幻觉——**"全量堆砌可用"不是配置属性而是模型属性**

#### 5.5 Token 效率 vs 临床可用性的权衡（核心章节）

**LLM 评审的评估偏好**（"结构 + 术语"）：
- 章节结构完整、术语合规、表单清单齐全 → 高分
- **骨架版（H4, 6K）能以最小代价满足** → LLM 层最优

**专家的评估偏好**（"临床实操深度"）：
- SOP 子场景覆盖（急诊、拒收、患者自采）→ 权重高
- 岗位资格具体到人员资质要求 → 权重高
- 记录表单在真实工作流中可追溯 → 权重高
- **模板版（G, 38K）的详细内容提供了 LLM 没有自主生成的临床细节** → 专家层最优

**权衡表**：

| 维度 | H4 (6K) | G (38K) | 主导评估者 |
|------|---------|---------|----------|
| 章节结构完整性 | ✅ | ✅ | LLM judge |
| 术语规范 | ✅ | ✅ | LLM judge |
| 临床子场景覆盖 | ❌ | ✅ | 专家 |
| 岗位资格具体性 | 中 | ✅ | 专家 |
| 记录可追溯性 | 中 | ✅ | 专家 |
| **token 开销** | **低** | **高** | 成本 |

**推论**：**不存在"单一最优"配置；存在"场景匹配的最优"配置**

#### 5.6 LLM judge 与专家分歧的机制

**可能解释**（待后续研究）：
- LLM judge 擅长验证"表面特征"（结构/术语/表单清单），因为这些在训练数据中模式密集
- 专家的"临床可用性"判断依赖实操经验，LLM 训练数据中很难学到（未观察到的不合规案例、未明示的行业惯例）
- Claude 比 GPT 与专家更一致（ρ=0.70 vs 0.30）——可能反映 Claude 在中文检验医学语料上的相对优势（**待验证**）

### 6. 讨论

#### 6.1 实证配置建议（对实验室的直接指南）

**核心消息**：**没有绝对的"最优配置"，只有"场景匹配的最优"**

| 使用场景 | 推荐配置 | Token | 依据 | 取舍 |
|---------|---------|-------|------|------|
| 快速生成初稿 / 内部迭代 | **H4_sop_only** 或 **E_rules** | 3–6K | LLM 层效率最优 | 需人工补充临床细节 |
| 多 SOP 编写 | H4 或 H3 | 6–13K | 任务特定骨架+规则 | 同上 |
| **正式文件 / CNAS 送审** | **G_template_rules** | **38K** | 专家层认可最高、跨模型稳健 | 较高 token 成本 |
| 完整质量手册 | F_template 或 G | 35–38K | 临床细节丰富 | 同上 |
| ❌ **避免** | **C_full** (71K) | — | GPT 生成下崩盘、跨评最差 | — |
| ❌ 避免 | A_bare / B_simple | — | 基线太低、缺合规性 | — |

**工作流建议**：**H4 生成初稿 → 专家人工校对补充 → 关键文件重新用 G 生成**（先用 H4 快速草拟，再用 G 精修）

#### 6.2 配置成分的作用机制总结

- **规则层**（术语禁令、模糊禁令）：无论 LLM 还是专家评估都有效，**必需**
- **骨架层**（章节/表单/角色结构）：LLM 层贡献明显，专家层也认可结构完整，**必需**
- **详细内容**（ISO 原文、详细表格）：LLM 层 Δ≈0，但**专家层揭示其对临床可用性有贡献**——ISO 原文非冗余
- **示例**（"⚠️ 张医生..."）：LLM 层方向性有害（照抄风险），专家层未单独验证——**建议剥离**

#### 6.3 LLM judge 自评偏见的实践警示

Claude +0.464 / GPT −0.472 的对称性证明偏见是**judge 端**的现象而非生成端的。实践中：
- **不要**用单一 LLM judge 做配置选择决策
- **应该**跨 LLM judge 交叉验证（或配合专家抽样校验）
- **骨架版配置**（E/H3/H4）在两个 judge 下偏见最小（<0.15）——在 LLM 层评估时相对可靠

#### 6.4 LLM judge 与专家分歧的实践警示

**不要**把 LLM judge 评分直接当真实质量——两个 judge 都系统性高估 0.55–0.93 分，且排序与专家不完全一致。

**应该**：
- **Claude judge 可作为专家"粗筛代理"**（ρ=0.51, ICC=0.55, n=3 raters）——初筛大批文件节省专家时间；但**绝对分需人工校准下移 ~0.9 分**
- **GPT judge 不适合作为专家代理**（ρ=0.33, ICC=0.22）——仅用于跨模型偏见检测
- **关键文件（送审/执行）必须专家终审**——尤其是 H4/C_full 生成的文件，LLM judge 高估最严重（Δ > 1）

#### 6.5 LLM 训练数据对配置研究的影响（探索性）

Claude 与专家一致度高于 GPT，可能与训练数据中中文医学/质量管理语料占比有关——但本研究无法直接验证。若此假设成立：
- **结论泛化性**：英语 QMS 文件生成可能有不同 config-optimal 发现
- **未来工作**：在英语 ISO 15189 文件、日语 ISO 15189 文件上复现

### 7. 局限性

- **7.1 专家样本 n=3 raters × 10 papers，全部来自同一医学实验室**（v16 更新）：
  - 组均值 n=1–2 篇/组，仍仅能支持排名方向性结论
  - **3 位专家间 ICC(2,k)=0.982 的极高一致性**可能**高估了客观性**，因为 3 位 rater 均为同单位 ISO 15189 内审员，共享专业文化、培训谱系、对模糊条款的解读习惯
  - 跨单位 3–5 位专家 × 30–50 篇的多中心验证是关键的未来工作
- **7.2 第一作者同时担任 Rater 1 与研究设计**：Rater 2/3 在不知晓 Rater 1 评分的情况下独立完成评分后才汇总；per-rater 数据完整公开，审稿者可独立判断结论是否依赖 Rater 1
- **7.2b 专家与 LLM 评审使用相同 5 维度**：存在 shared-measurement bias 风险（若评分维度对 LLM 友好，则 LLM 评审天然占优）；建议未来加入"临床可用性"等专家原生维度
- **7.3 GPT-5.4 生成仅 3 任务**：A1/B1/C1 81 篇，完整 15 任务覆盖预估 +270 篇待扩展
- **7.4 领域特异性**：ISO 15189 医学实验室限定，ISO 17025/ISO 9001 迁移性待验证
- **7.5 骨架化算法的主观性**：`strip_to_skeleton` 规则基于作者判断
- **7.6 SOP 选择依赖领域专家**：H4 任务→SOP 映射非自动化
- **7.7 第三方 judge 缺失**：仅 Claude/GPT，Gemini/open-source 未覆盖
- **7.8 语言限定**：全部中文
- **7.9 示例效应显著性待提升**：p=0.46 方向性发现，需更大样本验证

### 8. 结论（v15 重写——实证为主）

1. **最优配置取决于使用场景**：LLM 层（结构 + 术语效率）下 H4_sop_only (6K) 最优；专家层（临床实操深度）下 G_template_rules (38K) ≈ F_template (35K) 最优

2. **4 因子正交消融揭示配置成分的贡献方向**（LLM 层）：规则层必需（+0.511***）、骨架层必需（+0.213, p=0.055）、详细内容 LLM 层无贡献但专家层有贡献、示例方向性有害

3. **跨模型验证揭示"全量堆砌"的陷阱具模型依赖性**：C_full (71K) 在 Claude 生成下 LLM 层 4.59，在 GPT 生成下跨评 1.40–1.84 崩盘——"大配置也能用"是 Claude 特性非通用结论

4. **LLM judge 自评偏见**：Claude +0.464 / GPT −0.472，骨架版配置（E/H3/H4）偏见最小；跨评最客观排名下 G/H3/E/H4 统计等价

5. **3 位专家校验（ICC(2,k)=0.982）揭示 LLM judge 局限**：Claude judge 与专家均值排序中等一致（ICC(3,1)=0.548, p=0.04），GPT 评审相关性差（0.217, ns）；两者系统性高估 0.52–0.90 分；**H4 在 LLM 层 #1 在专家层 #5**（7 组中倒数第二）

6. **实践建议（场景分层）**：
   - 初稿/内部迭代 → H4 或 E_rules（3–6K）
   - 正式送审 → **G_template_rules** 或 F_template（35–38K）
   - 避免 → C_full（跨模型崩盘）、A_bare（基线太低）

7. **方法学启示**：LLM judge 可作为初筛工具但不可替代专家终审；跨 LLM judge 验证可诊断自评偏见；Python 合规评分 + 多层 LLM judge + 专家抽样是平衡效率与可信度的实用配置
